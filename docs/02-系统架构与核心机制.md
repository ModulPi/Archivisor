### 4.2 Agent 状态机

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| **IDLE** | 空闲，等待用户指令 | 初始状态/任务完成 |
| **PLANNING** | 规划中，拆解任务 | 收到用户指令 |
| **AWAITING_CONFIRM** | 等待用户确认 | 高风险操作，需审批 |
| **EXECUTING** | 执行中，调用工具 | 用户确认/低风险任务 |
| **AUDITING** | 审计中，记录操作 | 工具执行完成 |
| **DONE** | 任务完成 | 审计完成 |

### 4.4 降级模式说明

当 DeepSeek API 不可用时，系统自动进入降级模式：

| 层级 | 降级后行为 |
|------|----------|
| **第一层（规则匹配）** | 正常工作，不依赖 LLM |
| **第二层（意图识别）** | 降级为本地 Embedding 匹配（用 MiniLM 计算用户指令与 Skill 描述的相似度，阈值 > 0.75 时匹配） |
| **第三层（ReAct 推理）** | 不可用，提示用户「联网后可处理复杂任务」 |

### 4.5 事件驱动与决策模型的关系

> **事件驱动的处理链**（如文件新增自动归档、磁盘空间预警）不经过三层决策模型，**直接路由到规则引擎或 Skill 引擎**。三层决策模型仅处理**用户主动下发的对话指令**。

## 十、非侵入式系统集成方案

### 10.1 核心原则

> **Agent 不修改任何系统默认路径，不触碰注册表中的系统级配置，通过 Windows 符号链接实现透明数据重定向。**

### 10.2 核心技术：符号链接

符号链接是 Windows NTFS 提供的一种底层机制，在操作系统内核层面工作：

> **软件访问原路径时，Windows 内核自动、透明地将请求重定向到目标路径。软件完全察觉不到中间存在"转手"。**

| 类型 | 适用场景 |
|------|----------|
| **目录符号链接**（`mklink /D`） | 跨盘符目录重定向（推荐） |
| **目录交接点**（`mklink /J`） | 同盘符或跨盘符，功能更强（推荐用户目录） |

### 10.3 软件分类与目标路径

| 软件类别 | 示例 | Agent 目标路径 |
|----------|------|----------------|
| 即时通讯 | 微信、QQ、钉钉、飞书 | `D:\Data\IM\{软件名}\` |
| 浏览器 | Chrome、Edge、Firefox | `D:\Data\Browsers\{软件名}\` |
| 开发工具 | VS Code、IDEA、PyCharm | `D:\Data\DevTools\{软件名}\` |
| 游戏平台 | Steam、Epic、暴雪战网 | `D:\Games\Platforms\{平台名}\` |
| 创意工具 | Adobe 系列、OBS、DaVinci | `D:\Data\Creative\{软件名}\` |
| 效率工具 | Notion、Obsidian、Todoist | `D:\Data\Productivity\{软件名}\` |
| 虚拟机 | VMware、Docker、WSL | `D:\Data\Virtualization\{软件名}\` |
| 音乐/播客 | 网易云、Spotify | `D:\Media\Audio\{软件名}\` |

> 具体目标路径由 Agent 根据用户画像和顶层结构设计动态生成。

### 10.4 风险管控

| 风险场景 | 应对策略 |
|----------|----------|
| 原路径存在内容 | 迁移前检查，提示用户确认 |
| 目标路径已存在 | 检查是否为符号链接，提示用户选择处理方式 |
| 磁盘空间不足 | 迁移前检查剩余空间，不足时预警 |
| 软件正在运行 | 检测进程，提示用户关闭，或提供「关闭并继续」选项 |
| 目录被系统锁定无法移动 | 提供「重启后迁移」选项，将迁移任务写入 Windows 任务计划程序 |
| 权限不足 | 提示用户以管理员身份重启 Agent |
| 迁移中断 | 记录状态，支持断点续传和完整回滚 |
| 符号链接被误删 | 审计日志记录，Agent 可自动重建 |


## 十一、Skill 体系设计

### 11.1 什么是 Skill

**Skill 是预定义的"工作流模板"**——将一组原子工具的调用顺序、参数映射固化下来，形成可被 Agent 通过名称直接调用的"高级能力"。

| 对比维度 | 原子工具（Tool） | Skill（技能） |
|----------|-----------------|---------------|
| **粒度** | 单次操作 | 多步骤工作流 |
| **调用方式** | Agent 每次需规划调用顺序 | Agent 一次性调用，内部自动编排 |
| **LLM 参与** | 每次调用都需要决策 | 调用时无需 LLM 参与执行细节 |
| **适用场景** | 基础操作 | 高频重复的复合任务 |

### 11.2 原子工具与 Skill 的分层原则

> - **Agent 在第三层（ReAct 循环）中只能调用原子工具**，不能调用 Skill
> - Skill 是第二层的执行单元，不能被第三层作为子步骤调用
> - 这种设计避免递归调用导致的状态管理复杂度爆炸（Agent 调用 Skill → Skill 内部再调用 Agent → 无限循环）

### 11.3 Skill 匹配流程
用户指令
│
▼
提取关键词 → 精确匹配 Skill.keywords
│
├── 命中 → 直接调度该 Skill
│
└── 未命中 → 语义匹配（Embedding 相似度）
│
├── 相似度 > 0.75 → 调度最匹配的 Skill
│
└── 相似度 < 0.75 → 进入第三层（ReAct 推理）


### 11.4 Skill 沉淀流程（含优化器）
第三层执行成功 → 提取工具调用序列（含试错步骤）
│
▼
优化器（LLM + 预设规则）：
├── 去除试错步骤（如执行失败后重试的操作）
├── 合并同类操作（如连续多次 move_file 合并为批量移动）
└── 识别可参数化的变量（替换为 {user_profile}、{project_name} 等占位符）
│
▼
生成 Skill 草稿 → 通知用户确认
│
├── 用户确认 → 保存为正式 Skill（下次走第二层）
└── 用户拒绝 → 丢弃草稿，或记录为「用户不喜欢的模式」

### 11.5 示例 Skill 列表

| Skill 名称 | 分类 | 触发关键词 | 核心工具组合 |
|------------|------|-----------|-------------|
| `clean_wechat_cache` | cleanup | 微信、缓存、清理 | scan→filter→report→confirm→delete |
| `clean_downloads` | governance | 下载、归类、整理 | scan→classify→move→audit |
| `archive_project` | archive | 归档、完成、项目 | compress→move→update_index→audit |
| `find_duplicates` | search | 重复、去重、相同 | scan→hash→group→report |
| `setup_dev_project` | scaffold | 项目、初始化、脚手架 | create_structure→copy_template→update_index |
| `disk_cleanup_report` | analysis | 磁盘、空间、满了 | scan_disk→analyze→recommend→report |
| `redirect_app_data` | system | 软件、安装、重定向 | detect→identify→confirm→move→symlink |


## 十二、降级与容错设计

### 12.1 熔断与重试

| 机制 | 说明 |
|------|------|
| **熔断** | API 连续失败 N 次后暂停调用，进入降级模式 |
| **指数退避重试** | 失败后等待 1s→2s→4s→8s，逐步延长间隔 |
| **超时控制** | 单次 LLM 调用 30s 超时，超时后中断并降级 |

### 12.2 操作幂等性

| 场景 | 幂等保证 |
|------|----------|
| 文件移动 | 执行前记录源路径，失败后可重试，成功后再清理源 |
| Skill 执行 | 每个步骤标记执行状态，中断后可断点续传 |
| 规则触发 | 生成唯一 Job ID，防止重复触发 |


## 十三、可观测性设计

### 13.1 三大支柱

| 支柱 | 数据内容 | 用途 |
|------|----------|------|
| **Tracing（链路追踪）** | 用户指令从输入到输出的完整链路 | 排查慢操作、定位瓶颈 |
| **Metrics（指标采集）** | API 调用次数/成功率/延迟、Skill 热度 | 性能监控、成本分析 |
| **Logging（结构化日志）** | JSON 格式系统运行日志 | 故障排查、审计追溯 |

### 13.2 关键指标

| 指标类别 | 具体指标 |
|----------|----------|
| **性能指标** | API 响应时间 P50/P95/P99、Skill 执行平均耗时 |
| **成本指标** | 每日 LLM Token 消耗、预估费用 |
| **质量指标** | Skill 成功率、用户打断/重试率 |
| **系统指标** | 内存占用、CPU 使用率、数据库大小 |
| **业务指标** | 日均治理文件数、释放空间总量、Skill 热度排名 |

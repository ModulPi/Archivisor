## 五、数据模型

### 5.1 核心数据表

| 表名 | 说明 | 主要字段 |
|------|------|----------|
| `file_metadata` | 文件元数据索引 | id, path, name, extension, size, md5_hash, modified_at, is_deleted |
| `audit_log` | 操作审计日志 | id, operation_type, source_path, target_path, status, created_at |
| `audit_log_archive` | 审计日志归档（按季度分区） | 同 audit_log |
| `user_profile` | 用户画像 | id, occupation, work_type, tools, frequent_folders, preferences, rejected_schemes, accepted_rules |
| `governance_rules` | 治理规则 | id, name, condition, action, priority, is_active |
| `skills` | Skill 定义 | id, name, description, keywords, steps_json, usage_count, success_rate |
| `skill_embeddings` | Skill 向量索引 | skill_id, embedding_vector |
| `project_templates` | 项目模板 | id, name, type, structure_json, variables |
| `sessions` | 对话会话 | id, session_id, profile_id, start_time, message_count |
| `metrics` | 系统运行指标 | id, metric_name, metric_value, timestamp |
| `trace_logs` | 链路追踪日志 | id, trace_id, span_id, operation, duration, status |
| `migration_plans` | 符号链接迁移计划 | id, plan_name, source_path, target_path, status, scheduled_time |

### 5.2 冷热数据分离策略

| 数据类型 | 热存储 | 冷存储 | 迁移条件 |
|----------|--------|--------|----------|
| 文件元数据 | SQLite 主表 | 归档表 | 修改时间超过 30 天且未被访问 |
| 审计日志 | SQLite 审计表 | 季度分区表 | 创建时间超过 90 天 |

### 5.3 性能基准目标

| 数据规模 | 看板核心查询 | 文件搜索 |
|----------|-------------|----------|
| 10 万级文件 | < 500ms | < 300ms |
| 100 万级文件 | < 2s | < 1s |


## 六、技术栈

| 层级 | 技术选型 | 用途 |
|------|----------|------|
| **桌面框架** | Electron + React + TypeScript | GUI 界面，数据看板，对话窗口 |
| **UI组件库** | Ant Design / MUI | 加速 UI 开发 |
| **数据可视化** | ECharts / Recharts | 磁盘看板图表 |
| **后端框架** | FastAPI | REST API + WebSocket |
| **Agent框架** | LangGraph | Agent 状态机与流程编排 |
| **大模型** | DeepSeek API（主）+ 本地 MiniLM/SBERT（离线降级） | Agent 推理 + 本地 Skill 语义匹配 |
| **文件操作** | pathlib, shutil, psutil, watchdog | 文件系统交互与监控 |
| **本地数据库** | SQLite + sqlite-vec | 元数据存储 + 向量索引 |
| **日志** | structlog | 结构化日志 |
| **打包工具** | electron-builder + PyInstaller | 应用打包分发 |
| **Windows集成** | winreg, pywin32 | 注册表读取、符号链接管理 |
| **缓存** | 内存 LRU Cache | 看板数据、Skill 预加载缓存 |
| **事件驱动** | asyncio + 自定义 Event Bus | 异步事件分发 |


## 七、开发路线图

| 阶段 | 周期 | 核心目标 | 关键产出 |
|------|------|----------|----------|
| **第一阶段 1a** | 2 周 | 核心数据底座 | 元数据索引、审计日志、文件操作 API、数据看板、符号链接管理、迁移执行与回滚 |
| **第一阶段 1b** | 2 周 | 交互层与事件骨架 | 对话窗口 GUI、用户画像冷启动、软件安装感知、文件事件监听、结构化日志 |
| **第二阶段** | 4 周 | Agent 驱动的分析与设计 | 意图路由、任务规划、磁盘分析、方案生成、方案预览与确认、项目脚手架、画像持续学习 |
| **第三阶段** | 3 周 | 持续治理与自治 + Skill 体系 | Skill 引擎（含预加载）、规则引擎、自动路由、冷热数据分离、审计回滚、打包分发 |


## 八、关键技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| GUI 框架 | Electron + React | UI 表现力强，开发效率高，适合数据看板 |
| 后端语言 | Python | AI 生态丰富，与 Agent 框架无缝集成 |
| Agent 框架 | LangGraph | 状态管理、流程控制能力强 |
| 大模型 | DeepSeek API（主）+ 本地 Embedding 模型（降级） | 成本低、中文强；离线时支持 Skill 语义匹配 |
| 本地意图识别 | Embedding 相似度匹配（MiniLM/SBERT） | 无需训练分类模型，轻量可落地，模型 ~80MB |
| 本地数据库 | SQLite + sqlite-vec | 轻量级零配置，支持向量检索 |
| 前后端通信 | HTTP + WebSocket | API 调用 + 长任务进度推送 |
| 系统集成方式 | 符号链接（非侵入式） | 不修改系统路径，零兼容性风险 |
| 软件安装处理 | 先装后迁 | 避免拦截安装过程的兼容性问题 |
| 权限模式 | 监督 + 自治混合 | 高风险确认，日常自动化 |
| 跨平台策略 | 先专注 Windows | 用户基数大，文件管理需求普遍 |
| 处理模式 | 三层决策模型（规则→Skill→ReAct） | 成本可控、系统可生长 |
| 架构模式 | 事件驱动 + Agent-Centric | 解耦模块、支持异步任务 |
| 原子工具与 Skill 边界 | Agent 第三层只能调用原子工具，不能调用 Skill | 避免递归调用导致的状态管理复杂度爆炸 |


## 九、安全与隐私设计

| 原则 | 具体措施 |
|------|----------|
| **本地优先** | 所有数据存储在本地 SQLite，不上传云端 |
| **API Key 安全** | DeepSeek API Key 使用 Electron safeStorage 加密存储 |
| **操作可逆** | 删除优先使用回收站，支持一键回滚 |
| **权限隔离** | Agent 默认只操作用户目录，系统目录只读不写 |
| **用户知情** | 高风险操作需用户明确确认 |
| **审计追溯** | 所有操作记录完整审计日志 |
| **系统级保护** | 符号链接操作需管理员权限，不修改注册表系统配置 |
| **迁移完整性** | 数据迁移前后校验完整性，异常中断不损坏数据 |
| **操作幂等性** | 文件操作支持重试，不产生副作用 |
| **目录锁定处理** | 数据目录被系统锁定时，提供「重启后迁移」方案 |


## 十四、项目命名建议

| 名称 | 含义 |
|------|------|
| **Archivisor** | Archive + Advisor（归档顾问） |
| **Datalis** | Data + Atelier（数据工坊） |
| **Struct** | Structure + Agent（结构 + 智能体） |
| **Cleo** | Clean + Organize 的结合 |
| **Orac** | Organize + Archive 的结合 |

**推荐：Archivisor**——兼顾"归档"与"顾问"双重含义，与项目定位高度契合。
